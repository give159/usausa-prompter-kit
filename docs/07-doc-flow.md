# 特典｜実務ドキュメントの一連の流れ
## 要件定義 → 詳細設計 → 実装仕様 → 利用手順書

1つのツールが「望み」から「使える道具」になるまで、実務では4つの文書を通ります。この特典では、実在するツール（うさうさプロンプター v1.7C PRO・全2,220行）を題材に、**同じ機能が4文書でどう姿を変えるか**を追跡します。

掲載する数値・仕様は、すべて実物のソースコードと突合済みです。架空の例ではありません。

---

## 1. なぜ4つも文書が要るのか（WHY）

家を建てるとき、施主の「日当たりのいいリビングが欲しい」という望みは、そのままでは大工さんが作れません。間取り図になり、施工手順になり、最後に住まい方の説明書になって、はじめて人が住めます。

ソフトウェアも同じです。**望み → 仕組み → 部品と数値 → 使い方**。この4段階を飛ばすと、「思っていたのと違うものができた」が起きます。

| 文書 | 答える問い | 主役 | 言葉づかい |
|---|---|---|---|
| ① 要件定義書 | 何を・なぜ作るか | 依頼する人 | ふつうの日本語 |
| ② 詳細設計書 | どう作るか（仕組み） | 設計する人 | 図と表 |
| ③ 実装仕様書 | どの部品・どの数値で | 作る人 | 数値と条件 |
| ④ 利用手順書 | どう使うか | 使う人 | ふつうの日本語（に戻る） |

面白いのは、**①と④が同じ「ふつうの日本語」に戻る**ことです。真ん中の②③でいちばん専門的になり、また日常の言葉へ帰ってくる。この山型が、実務ドキュメントの形です。

## 2. 【核心】1つの機能を4文書で追跡する

ここがこの特典の目玉です。実際の機能3つを、4文書すべてで追いかけます。

### 追跡例A：「読み上げ速度の調整」

| 段階 | 記述内容 |
|---|---|
| **① 要件定義** | 「速さが自分に合わないと読みにくい。あとから直せるようにしたい」（利用者の望み） |
| **② 詳細設計** | FR-01 読み上げ速度の調整。範囲は 80〜600 字/分（10刻み、既定300）。設定は保存され次回も引き継ぐ |
| **③ 実装仕様** | `setCpm(v)` で設定。`clamp(値, 80, 600)` で範囲外を丸める。数値でない入力は300に。UI側は `<input type="range" min="80" max="600" step="10" value="300">`。変更後 `recalcSpeed()` で再計算し `saveSoon()` で保存 |
| **④ 利用手順書** | 「速さが合わなければ、速度のスライダーで調整してください（80〜600の間。最初は300が目安）。↑↓キーで10ずつ微調整もできます」 |

**読みどころ**：①の「合わないと読みにくい」という曖昧な望みが、②で「80〜600」という数字になり、③で「範囲外はclampで丸める」という具体的な守り方になり、④で「最初は300が目安」という親切な案内に戻っています。**同じことを4回、解像度を変えて言っている**のが分かります。

### 追跡例B：「原稿の履歴を残す」

| 段階 | 記述内容 |
|---|---|
| **① 要件定義** | 「原稿を書き換えて失敗したとき、前の状態に戻したい」 |
| **② 詳細設計** | FR-12 版履歴。最大12件を保持。超過分は最も古いものから自動削除。保存先は端末内のみ |
| **③ 実装仕様** | `commitVersion(msg)` で `state.versions.unshift({id, ts, text, chars, msg})` により先頭へ追加。直後に `if(state.versions.length>12) state.versions.length=12;` で切り詰め |
| **④ 利用手順書** | 「『保存』するたびに、その時点の原稿が記録されます（最新が上・最大12件）。いつでも前の版に戻せます」 |

**読みどころ**：②の「最も古いものから自動削除」が、③では「先頭に足して、長さを12にする」という2行のコードになっています。**日本語の1文が、コードの1〜2行に対応する**——この対応関係が見えると、設計書とコードが繋がって見えてきます。

### 追跡例C：「AI相談の安全性」

| 段階 | 記述内容 |
|---|---|
| **① 要件定義** | 「原稿は外に出したくない。でもAIに相談もしたい」（相反する2つの望み） |
| **② 詳細設計** | FR-19〜22 AI連携。正常動作はアーティファクト版でのみ。通常版では案内表示のみで、送信ボタンを押しても通信は発生しない。NFR-03 データは端末内保存のみ（AI相談使用時を除く） |
| **③ 実装仕様** | `isArtifactEnv()` がURLを判定（`file:` / `localhost` / `127.0.0.1` は false）。この判定は `updateAiBar()` の案内文表示と、`callAI()`（`fetch("https://api.anthropic.com/v1/messages")`）冒頭の送信ガードの両方に使われている。`sendAI()`自体は環境を問わず呼ばれるが、`callAI()`のガードで通常版では通信自体が発生しない |
| **④ 利用手順書** | 「『相談』ボタンは特定の環境でのみ正常に動作します。通常の使い方（ローカルで開いた場合）では案内が出て、送信ボタンを押しても通信は発生しません」 |

**読みどころ**：①の**相反する望み**を、②が「環境で分ける」という設計判断で解決しています。実務で最も難しく、最も価値があるのがこの部分です。

**Before/After（このツールで実際にあった話）**：以前のバージョンでは、`isArtifactEnv()` が実際にゲートしていたのは案内バーの表示だけで、送信自体（`callAI()`）は環境を問わず試みられていました。設計判断（②）と実装（③）がずれていた典型例です。現在は`callAI()`冒頭にも同じ`isArtifactEnv()`のガードを追加し、②で決めた設計判断どおりに③が動くよう修正しています。**「設計書に書いた」ことと「コードがそのとおりに動く」ことは別**——この食い違いに気づき、閉じられるかどうかも、コードを読む価値の1つです。

## 3. 各文書の実物ダイジェスト

### ① 要件定義書（何を・なぜ）

実物の目次構成です。

```
1. 目的と背景        … 困りごとと、解決後の姿
2. 提供形態・利用者   … 誰が・どんな環境で使うか
3. 用語             … この文書内での言葉の定義
4. 機能要件（FR）    … できてほしいこと。ID付きで
5. 非機能要件（NFR） … 速さ・安全・対応環境など、機能以外の条件
6. 制限値一覧        … 数値の上限下限をまとめて
7. スコープ外        … 今回やらないこと
```

**実務のコツ**：7の「やらないこと」を必ず書きます。書かないと作業が無限に膨らみます。実物では「複数人での同時編集」「音声認識による自動追従」「旧.doc形式の取り込み」を明確に対象外としています。

### ② 詳細設計書（どう作るか）

```
1. アーキテクチャ概要  … 全体の作り。ファイル構成と行マップ
2. 状態管理           … 覚えておく情報の一覧
3. 保存データスキーマ   … どんな形で保存するか
4. 主要処理フロー      … 代表的な動きの手順
5. 設計判断（記録）    … なぜその作りにしたか、の理由
```

**実務のコツ**：5の「設計判断の記録」が後で効きます。実物では「単一ファイル構成にした理由（配布の容易さ／トレードオフはファイル肥大）」を残しています。半年後の自分や後任者が「なぜこうなってる？」と悩む時間を消せます。

### ③ 実装仕様書（部品と数値）

```
1. 機能を1つずつ細かく  … F-01形式で番号を振り、数値まで
2. 入力と出力          … 何を入れたら何が出るか
3. 画面の部品リスト     … ボタン・入力欄ごとの動き
4. データの詳しい形     … 型・初期値・上限下限
5. エラー・例外        … うまくいかない時どうする
6. テストの観点        … どうなれば合格か
```

**実務のコツ**：6の「テストの観点」を書いておくと、そのままテストコードになります。実物のツールでは「80より小さい値を入れても80になる」という観点が、`assert.equal(clamp(79,80,600), 80)` というテスト1行に対応しています。

### ④ 利用手順書（どう使うか）

```
① これは何？      … 30秒で「自分に関係あるか」が分かる
② 準備するもの     … 始める前に必要なもの
③ 使い方（手順）   … 1手順1動作で、番号つき
④ 困ったとき      … よくあるつまずきの先回り
⑤ 連絡先         … それでもダメな時の逃げ道
```

**実務のコツ**：この順番は「読む人の不安が消えていく順」です。①で関係あると分かり、②で準備でき、③で操作でき、④で詰まっても大丈夫、⑤で最後の安心。順番を入れ替えると、途端に読みにくくなります。

## 4. 現場のリアル：この順番どおりには進まない

正直に書きます。実務では、①→②→③→④と一直線には進みません。

**よくある戻り**：③実装仕様を書いていて「この数値、根拠がない」と気づき、①要件定義に戻る。④手順書を書いていて「この操作、説明できないほど複雑だ」と気づき、②設計に戻る。**戻るのは失敗ではなく、正常な工程**です。むしろ、作った後に気づくより何十倍も安上がりです。

**現実的な運用**：小さいツールなら、①要件定義だけしっかり書いて、②③は作りながら埋め、④は完成後に書く——これで十分機能します。4文書は「完璧に埋めるノルマ」ではなく「抜け漏れを防ぐチェックリスト」だと考えてください。

**このツールの場合**：実は本書のツールも、先に動くものを作り、後から要件定義書・詳細設計書を整えました。順番が前後しても、最終的に4文書が揃っていて、内容が実物と一致していれば、実務としては成立します（ただし、内容が実物と食い違ったままなのは論外です）。

## 5. あなたの番：この流れを自分でやる

この特典で「流れ」が見えたら、次は自分のツールで書いてみてください。穴埋めシートを用意しています。

| やること | 使うもの |
|---|---|
| 何に使うか思いつく | 事例集キット（職種別のアイデア） |
| ①②③を書く | 3つの設計書キット（穴埋めシート＋記入サンプル） |
| ④を書く | 手順書作成キット（5ブロックの型） |
| 配布後の対応 | トラブル対応テンプレ集 |

いずれも穴埋め式で、プログラミングの知識は不要です。分からない欄は「未定」と書いて先へ進んでください——**分からないことを正直に「未定」と書けるのが、良い設計書の第一歩**です。

---

## 一言まとめ

- 実務ドキュメントは「望み → 仕組み → 部品と数値 → 使い方」の4段階
- 同じ機能を、解像度を変えて4回書く。①と④は同じ日常語に戻る
- 戻り作業は失敗ではなく正常な工程。完璧より、抜け漏れ防止
- 大事なのは4文書が揃うことより、**内容が実物と一致していること**
